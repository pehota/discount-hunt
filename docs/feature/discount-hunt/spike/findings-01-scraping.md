# SPIKE-01 Findings: Aldi Süd Germany Scraping

**Date**: 2026-07-13
**Probe question**: Can we extract weekly discount items from Aldi Süd Germany's public catalogue via plain HTTP + HTML/JSON parsing, or does it require a headless browser?

---

## Binary Verdict

**WORKS — requires headless browser (Playwright/Puppeteer)**

Plain HTTP is blocked by Akamai Bot Manager. Playwright with Chromium succeeds and exposes a structured JSON API. Data is rich and structured.

---

## Approach Required

**Playwright (headless Chromium) + API interception**

Two-step approach:
1. Playwright navigates to `https://www.aldi-sued.de/de/angebote.html` with a real browser UA and executes Akamai's JavaScript bot-management challenge automatically.
2. The page makes authenticated calls to `https://api.aldi-sued.de/` — responses are intercepted via Playwright's `page.on("response", ...)` handler. No manual cookie management or token extraction needed.

## Anti-Bot Observations

| Layer | Behaviour |
|-------|-----------|
| `www.aldi-sued.de` | Protected by **Akamai Bot Manager** (`server: AkamaiGHost`). All plain HTTP requests return `403 Access Denied` regardless of UA, request headers, or cookie state. |
| `api.aldi-sued.de` | Also behind Akamai. Direct `requests` calls return `403`. Requires session cookies generated by Akamai's JS challenge. |
| Playwright | Bypasses Akamai successfully — JS challenge executes in Chromium, session cookies are set, API calls succeed with HTTP 200. |
| UA sensitivity | Full Chrome UA + `sec-ch-ua` headers tested — still 403 on plain HTTP. Akamai uses JS fingerprinting, not header inspection. UA alone is insufficient. |
| Rate limiting | Not observed (single session, ~20 requests during spike). No IP block triggered. |
| CAPTCHA | Not encountered during headless Playwright session in this spike. |

## Workarounds Without Playwright — Ruled Out

- **Third-party aggregators** (MeinProspekt, Kaufda): return their own SPA HTML; no usable public API for Aldi Süd data found.
- **Mobile app / alternate API domain**: same Akamai guard; `api.aldi-sued.de` is blocked.
- **CDN-hosted catalogue** (`/content/dam/...`): 403.
- **sitemap.xml**: 403.
- **JSON-LD on page**: Only `BreadcrumbList` present — no `Product`/`Offer` structured data.

Playwright is the minimum required approach. No lighter path exists.

---

## Edge Cases

1. **`regular_price` often absent**: Only 12/54 items had `wasPriceDisplay` this week. Fresh produce ("NATUR LIEBLINGE") shows no regular price reference — Aldi does not advertise a "was" price for all promotions. Schema must allow NULL; savings tracker must handle it.

2. **`onSaleDateDisplay` = start date, not end date**: Field value is "Verfügbar ab DD.MM.YYYY" (available from). The offer-end date must be parsed from DOM once per session and applied to all items.

3. **`servicePoint` = store/region code**: `B384` observed. Munich stores may use different codes. Need to verify service point codes for Munich or identify a region-independent endpoint.

4. **Pagination gap**: Category shows `totalCount: 74` but spike captured 54 items in one scroll. Full extraction requires paginating with `limit=12&offset=N` or triggering lazy load via Playwright scroll. Not a blocker — just loop the API.

5. **Future offers visible now**: Items with `onSaleDateDisplay: "Verfügbar ab 16.07.2026"` are next-week offers already in the API. Scraper should filter by comparing start date vs today to separate current-week from upcoming items.

6. **Aktion badge vs wasPriceDisplay**: Two independent signals mark promotions. Use both: `badges[].items[].displayText == 'Aktion'` OR `price.wasPriceDisplay != null`.

---

## Design Implications for DESIGN Wave

1. **Playwright is a hard dependency** for Aldi Süd. Plan headless browser runtime in deployment. Chromium download ~114 MB. Schedule as a background job (cron/celery), not on-demand per request.

2. **API-first, DOM-second**: Use `page.on("response")` to intercept JSON API responses. Avoid CSS-selector scraping of the SPA — the DOM will shift; the API shape is more stable.

3. **Two-phase `valid_until`**: Parse offer-week end date from DOM once per session (regex on `WOCHENANGEBOTE ... – SA., DD.M.`), apply to all API items.

4. **NULL-tolerant `regular_price`**: Schema field nullable. Savings tracker computes savings only when both prices present.

5. **Service point discovery**: Add a step to resolve the Munich service point code from the category tree before scraping.

6. **Scrape volume**: 74 items/week for Wochenangebote. One Playwright session + one paginated API loop is sufficient. No streaming or parallelism needed.

7. **Scrape schedule**: Offers run Mon–Sat. Scrape Monday 06:00 CET for current week. Optional Thursday run to capture "Verfügbar ab Thursday" forward items.

8. **Akamai ToS risk**: Playwright bypasses bot protection. Mitigate: persistent Chromium profile (session reuse), random delays (2–5s), limit to one scrape/week. CAPTCHA may appear under repeated or rapid scraping — no mitigation for that case beyond manual intervention or a CAPTCHA solver service.

## Addendum: prospekt.aldi-sued.de endpoint

**Date**: 2026-07-13
**Probe URL**: `https://prospekt.aldi-sued.de/kw30-26-op/page/2-3/hotspots_data.json`

### Plain HTTP — No Playwright Needed

**Yes, plain HTTP works.** No bot protection, no JS challenge, no Akamai on this domain.

```
curl -s "https://prospekt.aldi-sued.de/kw29-26-op-mp/page/4-5/hotspots_data.json"
→ HTTP 200, valid JSON
```

This is a Publitas-hosted digital catalogue viewer. The `hotspots_data.json` endpoints are served as static JSON files with no authentication. Standard `fetch`/`curl` with any User-Agent returns 200.

### Slug Discovery

The current week's slug is discoverable via a single HTTP redirect — no HTML parsing, no JS:

```
HEAD https://prospekt.aldi-sued.de/
→ 302 Location: //prospekt.aldi-sued.de/kw29-26-op-mp/
```

The slug format is `kw{week}-{yy}-op-mp` (KW = Kalenderwoche, mp = main prospekt). The plain 302 Location header contains the slug. This is deterministic and requires zero scraping.

The `kw30-26-op` slug the user found is a **different, shorter catalogue** (special offers subset, not the main weekly prospekt). The main prospekt follows the `-op-mp` suffix pattern.

### Catalogue Structure

The catalogue (KW29 tested) has **30 pages**, structured as double-page spreads. Each page pair `N-(N+1)` has a `hotspots_data.json` endpoint returning an array of hotspot objects. Only `type: "product"` hotspots carry product data; other types include `pageReference`, `externalLink`, `image`.

Page iteration: `page/1-2`, `page/2-3`, ... `page/30-31`. Pages beyond the catalogue return 404.

**Product distribution across KW29 (30-page catalogue):**
- Pages 1-2: cover/intro, 0 products
- Pages 2-3 through 27-28: product pages, 9–16 products per spread
- Pages 28-30: outro/back matter, 0 products

Total unique products found: **155 items** across KW29.

### was_price Coverage

| Metric | Value |
|--------|-------|
| Total products | 155 |
| Has `price` (regular) | 153 (99%) |
| Has `discountedPrice` (sale) | 31 (20%) |
| Has **both** prices | 31 (20%) |

The `price` field is the **regular/was price**. `discountedPrice` is the sale price. When a product is NOT on discount, only `price` is present (no `discountedPrice`). When both are present, the savings can be computed directly.

The 20% with-sale-price rate means this endpoint does **not** add coverage for the 80% of weekly items that are simply listed at their standard weekly price — those have a `price` but no `discountedPrice`. This matches the api.aldi-sued.de SPIKE-01 finding (12/54 = 22% had `wasPriceDisplay`). The coverage rate is essentially identical across both endpoints.

**Important distinction**: `price` here is NOT a permanent "normal" price. These are all weekly offer prices (Wochenangebote). The `price` field on a non-discounted item is the weekly offer price, not a stable baseline. A future scrape of the same SKU next week may show a different `price`.

### Valid-Until Date

**Not directly available in the hotspot JSON.** The `customLabel1` gives valid-from date only (`"13.7."`). The catalogue week end date (Saturday) must be inferred: `valid_from_date + 5 days`. Since KW29 starts Monday 13.7., end date = Saturday 18.7. This matches the api.aldi-sued.de SPIKE-01 data. No DOM scraping needed — simple date arithmetic from the slug's KW number suffices (ISO 8601 week arithmetic).

### Comparison vs api.aldi-sued.de

| Dimension | `api.aldi-sued.de` (SPIKE-01) | `prospekt.aldi-sued.de` (this addendum) |
|-----------|-------------------------------|------------------------------------------|
| Access method | Playwright required (Akamai blocks plain HTTP) | **Plain HTTP — no Playwright** |
| Item count | 54 items captured (74 in category) | **155 items** (full catalogue) |

| Per-kg price | `comparisonDisplay` (string) | `customLabel4` + `customLabel9` |
| Brand | In `brandName` | In `brand` |

### Verdict

**This endpoint supersedes `api.aldi-sued.de` for the discount-hunt use case.**

Reasons:

1. **No Playwright dependency.** Plain HTTP. Eliminates the Chromium 114 MB runtime, cron scheduling complexity, and CAPTCHA risk entirely.
2. **More items.** 155 vs 54 (2.9× more coverage). The prospekt includes all weekly items; the api.aldi-sued.de Wochenangebote category was incomplete.
3. **Regular price on 99% of items** via `price` field, vs 22% via `wasPriceDisplay` on the API. This is a fundamentally better field for tracking price history.
4. **Simpler slug discovery.** One HTTP redirect. No session management.
5. **National catalogue.** No store/service-point dependency.

The only loss vs `api.aldi-sued.de`: the `discountedPrice` rate (20%) is nearly identical to the `wasPriceDisplay` rate (22%) — no improvement on discount detection coverage. However, the `price` field now gives us a baseline price for 99% of items, which is more valuable for a price-comparison tracker than the "was" price on 22% of items.

**Recommended approach for implementation**: use `prospekt.aldi-sued.de` as primary source. Playwright / `api.aldi-sued.de` is no longer needed for Aldi Süd.
